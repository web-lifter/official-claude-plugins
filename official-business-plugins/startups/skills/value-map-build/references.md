The value map (left half of the Value Proposition Canvas) per Strategyzer. See [`startups/SOURCES.md`](../../../SOURCES.md) for full citations.

## The Value Proposition Canvas

- **Osterwalder, Alexander; Pigneur, Yves; Bernarda, Greg; Smith, Alan.** *Value Proposition Design.* Wiley, 2014.
  - Canonical reference. The value map has three components — products & services, pain relievers, gain creators — that map onto the customer profile's jobs, pains, and gains. The mapping is the substance of the canvas; an unmapped reliever is a vanity feature.

## Why every reliever maps to a prioritised pain

Strategyzer's discipline is that the canvas only contains *elements that connect*. A pain reliever that doesn't reduce or remove a specific prioritised pain is either misnamed or premature. The skill enforces this by failing fit-check on unmapped relievers; the user is forced to either drop the reliever, name the pain it addresses, or add the pain to the profile.

## Why mechanisms must be observable

"Easier" / "faster" / "better" are aspirations, not mechanisms. A mechanism is the *specific verb* by which the reliever acts — "auto-reconciles transactions from the bank feed in real time" is a mechanism; "makes accounting easier" is not. Without observable mechanisms the value map cannot drive an MVP scope.

## Why VPC is per-segment, per-version

Multi-segment ventures have one VPC per segment because relievers and creators are segment-specific. Each VPC is versioned because the value proposition is expected to evolve as discovery and validation produce new evidence. The version number plus the `## Diff from v<N>` section in subsequent versions gives the audit trail Ries's validated-learning loop requires.
